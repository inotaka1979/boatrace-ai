## 0. エグゼクティブサマリ

現状は Lighthouse A+ / iOS PWA 致命バグ修正済みの安定リリース段階だが、**6専門家の独立レビューを突き合わせた結果、本質的な伸びしろは「予測精度（ROI）」と「データ層の構造化」に偏在**している。UX とコード品質は既に高水準で、追加投資のROIが逓減し始めている。

- **次1ヶ月の最優先軸**: ROI軸への KPI 切替 ＋ 場別×級別×コースの3次元勝率テーブル整備
- **次3ヶ月の構造軸**: localStorage→IndexedDB 移行・スキーマバージョニング・特徴量パイプライン抽出
- **継続的な堅牢化**: VRT / iOS実機 CI / Bundle size budget / Lighthouse CI のゲート化

P0 を 11 件、P1 を 18 件、P2 を 8 件に整理した。重複指摘（複数の専門家が独立に指摘）は信頼度が高いため最優先扱いとする。

## 1. レビュアー別の核心指摘（要約）

| 役割 | コア結論 | 最重要1件 |
|------|----------|-----------|
| 設計者A（UX） | 5画面構成は良好。詳細画面の情報密度過多と色覚配慮不足、PWA再起動時の状態復元が穴 | 詳細画面のTabs化（出走表/AI根拠/オッズ&買い目） |
| 設計者B（データ） | スクレイプ運用は堅実。localStorage の上限・スキーマ管理・retention policy が未体系 | localStorage→IndexedDB 移行＋schema_version 導入 |
| コーダー | コードは「動いている」が `app.js` 5500行モノリス、Pythonスクレイパに重複パターン | scrape_results.py:215 の atomic_write_json 統一（1行） |
| QA-A（性能/A11y） | Perf 85達成済み。残課題は VRT / iOS実機 / scheduler API / 色覚 | Lighthouse CI ゲート（Perf≥85, LCP≤3s, CLS≤0.1, A11y≥95） |
| QA-B（セキュリティ） | PAT撤去・atomic write・safe_storage の多層防御が機能。残るのは外部API依存とCSP `unsafe-inline` | API スキーマ検証＋停止時フォールバックダイアログ |
| 競艇プロ | 構造は秀逸だが「場別×級別×コース3次元化」「決まり手シナリオ統合」が手薄 | KPI を「的中率」から「ROI（EV基準買い目絞り）」に転換 |

**6名のうち3名以上が独立に指摘した重複論点**:
1. localStorage の容量上限・スキーマバージョニング（設計者B / QA-B / コーダー）
2. iOS standalone PWA での状態復元・致命バグ再発防止（設計者A / QA-A / QA-B）
3. CI/品質ゲート不足（Lighthouse / VRT / mypy / size budget）（コーダー / QA-A / 設計者B）
4. EV/ROI ベースの買い目絞り（競艇プロ / 設計者B が「予測層責務分離」で同方向）

## 2. P0（次スプリント＝1〜2週で着手）

### P0-1【予測精度】場別×級別×コース 3次元勝率テーブルを stadiumDB に追加（競艇プロ A-01）
- **現状**: `getStadiumCourseWinRate(sid, course)` は場×コースの2次元、級別補正は乗算のみ
- **改善**: `stadiumDB[sid].courseWinRateByGrade = {1:{A1:0.55,A2:0.43,B1:0.38,B2:...}, ...}` を `build_db.py` で集計、predict 側で `log_odds(p)` ベース化
- **効果見込み**: 的中率 +8〜12%
- **影響ファイル**: `scripts/build_db.py`, `assets/app-rest.js`（scoreBoatV2）

### P0-2【予測精度】決まり手シナリオ動的統合（競艇プロ A-02）
- **現状**: 隣接艇の sashi/makuri 率を加点するだけで、レース全体の「逃げ成功確率」を確率モデル化していない
- **改善**: `nige_success = ∏(1 - p_attack(others))` をレース毎に算出し log_odds 化
- **効果見込み**: 回収率 +6〜10%
- **影響ファイル**: `assets/app-rest.js` の predictRace 内（A〜H 8カテゴリの上位に scenario レイヤを新設）

### P0-3【予測精度】KPI を「的中率 ≥ 12%」から「平均EV ≥ 1.20 かつ 30日ROI ≥ 110%」へ再定義（競艇プロ）
- **現状**: 設定に「点数」しかなく、点数固定で運用
- **改善**: race_type（本命/混戦/穴）別に推奨点数と evMin を動的化。設定UIに「ROI優先 / 的中優先 / バランス」プリセット
- **影響**: 設定画面（pageSettings）、`generateBetsV2()` シグネチャ拡張、成績トラッカーのKPI表示更新

### P0-4【UX】レース詳細画面の Tabs 化（設計者A P1）
- **現状**: 縦スクロール 10画面超、A〜H 8カテゴリが折返し多発
- **改善**: 「出走表」「AI根拠」「オッズ＆買い目」の3タブ。AI根拠はカテゴリ別アコーディオン＋影響度上位3つのみデフォルト展開
- **影響**: `index.html` detailPrediction div、`assets/app-rest.js` の openRace（L2500〜3100）、`renderPredictionHtml`

### P0-5【UX】iOS PWA 再起動時の状態復元（設計者A P0）
- **現状**: SW skipWaiting後にトップへリセットされる
- **改善**: `sessionStorage` に `_lastPage / _lastStadium / _lastRace` を保存、起動時に showPage 前に復元
- **影響**: `assets/app-critical.js` boot シーケンス

### P0-6【データ】localStorage スキーマバージョニング & マイグレーション（設計者B / QA-B / コーダー が共通指摘）
- **現状**: `boatrace_*` キーが固定ハードコード、A+化で `boatrace_featurestats / boatrace_platt` 追加されたが移行ロジック皆無
- **改善**: `boatrace_schema_version` キー導入、起動時 `migrations[]` を順次実行。30日経過の重みは reset 候補化
- **影響**: `src/utils/safe_storage.js` に migration runner 追加、`assets/app-critical.js` boot

### P0-7【セキュリティ】API スキーマ検証＋停止時フォールバックUI（QA-B P1-1）
- **現状**: 非公式API停止で `{error:'offline'}` 503 が返り、UI が空白に
- **改善**: 期待schema hash チェック、失敗時に「APIサービス停止」モーダル（再試行 / キャッシュ使用 / GitHub Issues link）
- **影響**: `sw.js` fetch ハンドラ、`assets/app-critical.js` のローダ

### P0-8【コード品質】`scripts/scrape_results.py:215` の atomic_write_json 統一（コーダー P0-3）
- **現状**: 直接 `open(OUTPUT,"w")` で書込、kill/disk-fullで corrupt JSON 残存
- **改善**: 1行修正 `atomic_write_json(OUTPUT, output)`
- **工数**: 5分

### P0-9【性能】Lighthouse CI ゲート（QA-A）
- **現状**: Lighthouse は手動計測のみ
- **改善**: `.github/workflows/lighthouse.yml` で PR毎計測、しきい値 `Perf≥85 / LCP≤3s / TBT≤300ms / CLS≤0.1 / A11y≥95`、breach で fail
- **影響**: GitHub Actions, `.lighthouserc.json` 新設

### P0-10【UX/A11y】色覚多様性配慮（race-type label）（設計者A P2 → 多数票でP0格上げ）
- **現状**: `.type-honmei`(青) `.type-ana`(赤) は赤緑色盲で識別困難
- **改善**: 絵文字 ⚡📊🔥 ＋ 背景パターン（repeating-linear-gradient の斜線）の二重化
- **影響**: `index.html` CSS

### P0-11【堅牢性】Web Worker 無応答時の main thread fallback（QA-A P0-2）
- **現状**: Worker 例外で main 側 Promise が永遠待機
- **改善**: `Promise.race([workerCall, timeoutFallback(3000)])` パターン、main 側に同期版 predict をフォールバック実装
- **影響**: `assets/app-rest.js`（predictRace 呼出箇所）

## 6. リスクと意思決定ポイント

| リスク | 緩和策 |
|--------|--------|
| IndexedDB 移行で既存ユーザーの予測重みが失われる | migration runner に backup→restore 経路、失敗時 localStorage に rollback |
| KPI を ROI 軸に変えたら短期的に的中率が下がって見える | UI に「的中率 / 平均EV / ROI」を3軸表示、「期待値理論で長期回収を狙う」旨をヘルプ明記 |
| Tabs 化で既存ユーザーが情報を見つけられない | 初回 Tabs 切替時にツールチップ、「以前の縦長表示」復元設定を1ヶ月だけ提供 |
| 非公式API仕様変更 | schema hash 検証＋fallback ダイアログ＋手動 GitHub Issue 通知 |
| Lighthouse CI が flaky で PR が頻繁に block | 同一PRで3 run の中央値で判定、しきい値に ±5 のバッファ |

## 7. メトリクス

### 開発メトリクス
- バンドルサイズ: critical / rest / worker_predictor の budget 設定値以下
- テストカバレッジ: ユニット 100+ ＋ VRT ＋ iOS実機 CI green率 ≥ 95%
- Lighthouse 4指標（Perf/A11y/BP/SEO）すべて ≥ 95

### プロダクトメトリクス（KPI再定義）
- **主指標**: 30日 ROI（推奨買い目だけを購入したと仮定したバックテスト）≥ 110%
- **副指標A**: 平均 EV（推奨買い目）≥ 1.20
- **副指標B**: 的中率（参考値）— 下がっても OK、ROI が上回れば成功
- **副指標C**: race_type 別 ROI（本命 / 混戦 / 穴）— 穴での過剰投資を防ぐガード
